# Plan 2026-01-21 — Gotify v2

**Date/Heure**: 2026-01-21
**Version**: v2

## Objectif

Envoyer une notification sur un serveur Gotify en cas de succès ou d'erreur lors de l'exécution du système.

## Étapes

1. **Créer un module de notification Gotify**
   - Créer un fichier `src/gotify.rs` pour le client Gotify
   - Implémenter une struct `GotifyClient` avec les champs : `url: String`, `token: String`
   - Implémenter une méthode `send_notification(&self, title: &str, message: &str, priority: u8) -> Result<(), Box<dyn std::error::Error>>`
   - Utiliser `reqwest::blocking::Client` (déjà présent avec features `blocking` et `json`) pour envoyer un POST vers `{url}/message` avec le header `X-Gotify-Key`

2. **Définir la configuration**
   - Lire les variables d'environnement :
     - `GOTIFY_URL` : URL du serveur Gotify (ex: `https://gotify.example.com`)
     - `GOTIFY_TOKEN` : Token d'application Gotify
   - Créer une fonction `GotifyClient::from_env() -> Option<GotifyClient>` qui retourne `None` si les deux variables ne sont pas définies

3. **Déclarer le module dans main.rs**
   - Ajouter `mod gotify;` en haut de `src/main.rs`

4. **Intégrer les notifications aux points de succès/erreur**
   - Créer le client Gotify optionnel au démarrage avec `GotifyClient::from_env()`
   - Après le bloc `if response.status().is_success()`, envoyer une notification de succès (priorité 5, titre "1min.ai - Succès")
   - Dans le bloc `else`, envoyer une notification d'erreur (priorité 8, titre "1min.ai - Erreur")
   - Ignorer silencieusement les échecs d'envoi avec `if let Err(e) = ... { eprintln!("...") }`

5. **Mettre à jour la documentation**
   - Ajouter `GOTIFY_URL` et `GOTIFY_TOKEN` dans `.env.local.example`

## Fichiers à toucher (scope)

- `src/gotify.rs` (nouveau) — Client Gotify en Rust
- `src/main.rs` — Déclaration du module + intégration des appels de notification
- `.env.local.example` — Documentation des variables d'environnement

## Risques

| Risque | Mitigation |
|--------|------------|
| Serveur Gotify indisponible | Timeout court (5s via `.timeout()`) + ne pas bloquer l'exécution principale |
| Token invalide | Log d'avertissement via `eprintln!`, pas de panique |
| Dépendance HTTP | Utiliser `reqwest` déjà présent dans Cargo.toml (features: blocking, json) |
| Erreur réseau | Encapsuler dans un `Result`, ignorer silencieusement les échecs |
